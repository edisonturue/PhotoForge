# 技能方案：全面推演器（deep-think）

---

## 一、定位

**用户输入一句话，技能自动推演出完整的全貌。**

适用：代码功能、产品决策、学习规划、生活选择、任何涉及多个因素需要想全的事。

不适用：简单事实性问题、已有明确方向只需执行的任务。

---

## 二、触发

```
/deep-think [你的问题或需求]
```

---

## 三、执行流程

### Step 1：意图诊断与复杂度判断（合并）

一步完成两件事：理解用户意图 + 判断是否值得推演。

```
你说了：[原话]
你真正想要：[推断]
复杂度：高 / 中 / 低
```

**复杂度判断标准：**

| 级别 | 信号 | 处理方式 |
|------|------|----------|
| 高 | 涉及多个相互影响的因素；有多种路径需要选；做错了代价大；用户可能遗漏重要方面 | 走完整推演 |
| 中 | 有一定复杂度但方向较明确；需要考虑一些关联方面 | 走简化推演（跳过路径对比，直接给推荐 + 关键注意事项） |
| 低 | 有唯一正确答案；用户已想清楚只需执行；改动范围很小 | 直接回答，不走推演 |

如果推断和原话差距较大，输出推断后问用户确认。

### Step 2：上下文加载

| 问题类型 | 需要什么 | 来源 |
|----------|----------|------|
| 代码 | 架构、实体清单、已有功能、编码约定 | 自动读 CLAUDE.md / .cursorrules |
| 产品 | 背景、目标用户、竞品、约束 | 对话中提取 |
| 学习 | 当前水平、目标、可用时间 | 对话中提取 |
| 生活 | 核心诉求、约束条件、优先级 | 对话中提取 |

**规则：** 上下文不足时，在推演结果开头列出"以下推演基于有限信息，我需要知道但你没告诉我的：[列表]"。不阻塞流程，让用户看完推演后再补充。

### Step 3：关联扩展（核心）

根据问题类型执行对应的检查清单，将结果分为三类输出：

**检查清单：**

```
代码开发：
□ 涉及哪些数据实体？CRUD 完整吗？
□ 搜索/筛选/排序需要覆盖吗？
□ 导出需要支持吗？
□ i18n 需要新增吗？
□ 哪些已有组件/页面需要联动修改？
□ 快捷键需要新增/更新吗？
□ 空状态怎么处理？
□ 加载状态怎么处理？
□ 错误状态怎么处理？

产品决策：
□ 目标用户是谁？他们真的需要吗？
□ 竞品怎么做的？
□ 对现有体验有什么影响？
□ 技术成本和维护成本？
□ 有没有更轻量的替代方案？
□ 做了之后下一步会想做什么？

学习规划：
□ 学了要用来干什么？
□ 有没有更合适的替代选项？
□ 怎么分阶段？
□ 常见放弃节点在哪？
□ 需要什么前置知识？
□ 高质量资源有哪些？

生活选择：
□ 可逆性如何？
□ 最坏情况是什么？能承受吗？
□ 会影响哪些人？
□ 有没有折中方案？
□ 短期 vs 长期影响分别是什么？
□ 有遗漏的隐性成本吗？
```

检查结果分三类输出：

| 类别 | 含义 | 示例 |
|------|------|------|
| 没想到的事 | 用户没提到但需要做的事/考虑的因素 | "搜索需要覆盖智能相册名称" |
| 边界情况 | 容易遗漏的异常/极端场景 | "条件为空时显示什么？" |
| 关联影响 | 对已有功能/生活的连带影响 | "Sidebar 筛选逻辑需要区分" |

**方向评估（仅当存在明显更优路径时输出）：**

```
你的方向：[从用户输入推断]
更优路径：[具体方案]
为什么更好：[理由]
```

如果用户的方向已经合理，不输出此项。

### Step 4：方案推荐

根据复杂度决定方案数量：

| 复杂度 | 方案数量 | 输出方式 |
|--------|----------|----------|
| 高 | 2-3 个 | 完整对比表 |
| 中 | 1-2 个 | 简要对比 |
| 低 | 1 个 | 直接推荐 |

对比表的列根据场景灵活调整：

```
代码场景常用列：思路 / 优点 / 缺点 / 工作量
日常场景常用列：思路 / 优点 / 缺点 / 风险 / 适合
```

推荐时评估 6 个维度：

| 维度 | 代码 | 日常 |
|------|------|------|
| 一致性 | 与已有架构一致？ | 与长期目标一致？ |
| 侵入性 | 改动多少文件？ | 影响多大范围？ |
| 可逆性 | 容易回退？ | 能反悔？ |
| 即时价值 | 用户能立即感受到？ | 能立即受益？ |
| 依赖链 | 有前置任务？ | 有前置条件？ |
| 风险 | 会破坏已有功能？ | 最坏能承受？ |

### Step 5：风险预警

```
🔴 高风险：[可能导致失败或不可逆损失]
🟡 中风险：[可能导致体验不佳或需要返工]
🟢 低风险：[小问题，但值得知道]
⚪ 已知取舍：[选了 A 就不能做 B]
```

没有风险就不输出此项。不要为了凑数而编造风险。

### Step 6：下一步（推演之后做什么）

推演完成后，告诉用户接下来可以做什么：

```
代码场景："确认方案后，我可以开始执行。需要我先输出实现步骤拆解吗？"
日常场景："以上是全面分析，最终决定权在你。需要我深入展开某个方面吗？"
```

### Step 7：变更报告（执行阶段必做）

当用户确认方案、AI 开始执行后，每次操作文件都必须在结尾输出变更报告：

```
本次变更：
- 新增：[文件路径] — [一句话说明做了什么]
- 修改：[文件路径] — [改动了什么]
- 删除：[文件路径] — [为什么删除]
```

规则：
- 每次有文件操作就必须输出，不管改动大小
- 不要等用户问"你改了什么"才说
- 如果一次操作涉及多个文件，全部列出
- 如果没有文件操作（纯分析/讨论），不输出此项

---

## 四、输出格式

根据复杂度自动调整详细程度。以下是完整格式（高复杂度），中/低复杂度省略对应部分。

```markdown
## 推演

**你说了：** [原话]
**你真正想要：** [推断]

### 你可能没想到的事
1. [关联方面 1]
2. [关联方面 2]

### 边界情况
- [边界 1]
- [边界 2]

### 关联影响
- [影响 1]
- [影响 2]

### 方向评估（如有更优路径）
你的方向：[推断]
更优路径：[方案]，原因：[理由]

### 路径对比（高复杂度用表格，中复杂度用文字）
| | 路径 A | 路径 B |
|---|---|---|
| [列1] | | |
| [列2] | | |

**推荐：** [X]，原因：[理由]

### 风险预警（没有则省略）
🔴 / 🟡 / 🟢 / ⚪

### 下一步
[告诉用户接下来可以做什么]
```

## 五、使用示例

### 示例 1：代码（高复杂度）

```
/deep-think 加智能相册

## 推演
你说了：加智能相册
你真正想要：让相册支持按条件自动填充照片，减少手动管理

### 你可能没想到的事
1. 搜索功能需要覆盖智能相册名称
2. 导出需要支持按智能相册导出
3. 侧边栏需要区分普通相册和智能相册
4. library.json 结构需要扩展
5. i18n 需要新增中英文文案

### 边界情况
- 智能相册的条件为空时显示什么？
- 条件编辑器加载失败怎么处理？
- 如果筛选条件匹配到 0 张照片，相册内容页显示什么？

### 关联影响
- 普通相册的列表组件需要适配两种类型
- Sidebar 的筛选逻辑需要区分
- 导出流程需要新增"按智能相册导出"选项

### 路径对比
| | A：独立实体 | B：扩展 Collection | C：筛选保存 |
|---|---|---|---|
| 思路 | 新增 SmartCollection 实体 | 给 Collection 加 isSmart 字段 | 把筛选条件存为"视图" |
| 优点 | 职责清晰，扩展性好 | 改动最小 | 最轻量 |
| 缺点 | 改动大 | 老数据需迁移 | 功能受限 |
| 工作量 | 大 | 中 | 小 |

**推荐：** A，原因：扩展性最好。你的项目还在早期，改动成本可控。方案 B 虽然改动小，但把两种不同语义的实体混在一起，后续维护会越来越痛苦。

### 风险预警
🟡 中风险：数据结构变更需向后兼容，老用户升级后 library.json 需要自动迁移
🟢 低风险：UI 变更范围可控

### 下步
确认方案后，我可以输出详细的实现步骤拆解（数据层 → UI层 → 集成 → 验证），你确认后再开始写代码。
```

### 示例 2：日常（高复杂度）

```
/deep-think 要不要辞职去创业

## 推演
你说了：要不要辞职去创业
你真正想要：在职业发展中找到更大的成长空间和自主权

### 你可能没想到的事
1. 经济缓冲：创业初期 6-12 个月可能没收入，你的存款能撑多久？
2. 家庭影响：家人的态度和支持度直接影响你的心理状态
3. 技能可迁移性：现有技能在创业方向上能直接用吗？
4. 机会成本：在职期间能不能先用业余时间验证想法？
5. 心理准备：创业的孤独感和不确定性远超打工

### 边界情况
- 如果创业失败，你的行业人脉和技能能让你顺利回到职场吗？
- 家庭是否有人依赖你稳定的收入（房贷、子女教育）？
- 你的健康状况能承受创业初期的高压吗？

### 关联影响
- 健康：创业初期的高压和不规律作息
- 关系：时间分配会大幅变化，家人可能需要额外沟通
- 财务：需要提前规划至少 6 个月的现金流

### 方向评估
你的方向：直接辞职创业
更优路径：先在职期间用业余时间做 MVP 验证，有初步收入后再全职。
原因：降低风险，验证市场需求，保留经济安全网。

### 路径对比
| | A：直接辞职 | B：在职验证 | C：找合伙人 |
|---|---|---|---|
| 思路 | 立即全职投入 | 业余时间做 MVP | 找互补的人分担 |
| 优点 | 全力以赴，速度快 | 风险最低 | 技能互补 |
| 缺点 | 风险高，经济压力大 | 精力分散，速度慢 | 决策复杂度增加 |
| 风险 | 高 | 低 | 中 |
| 适合 | 有 12 月+缓冲且想法已验证 | 想法还在探索期 | 需要你不具备的能力 |

**推荐：** B，原因：除非你有 12 个月以上的经济缓冲且想法已经过初步市场验证，在职验证是最稳妥的起步方式。很多成功的创业者都是从 side project 开始的。

### 风险预警
🔴 高风险：没有经济缓冲就辞职
🟡 中风险：创业方向未经市场验证
⚪ 取舍：选 B 意味着速度慢，但安全性高

### 下步
你可以先花一个周末做一个简单的市场调研（目标用户访谈、竞品分析），然后再决定要不要继续深入。需要我帮你设计调研方案吗？
```

### 示例 3：中复杂度（简化输出）

```
/deep-think 给照片卡片加 hover 效果

## 推演
你说了：给照片卡片加 hover 效果
你真正想要：提升照片浏览时的交互反馈

### 你可能没想到的事
1. 选中状态的样式需要和 hover 状态协调
2. 移动端没有 hover，需要考虑触屏反馈
3. 大量照片时 hover 效果不能导致卡顿

### 边界情况
- 照片缩略图还在加载时 hover 怎么处理？

### 关联影响
- CompareView 的照片卡片也需要同步加 hover

**推荐：** 阴影提升 + 轻微放大（scale 1.02），transition 150ms。这是最常见且不会干扰选中状态的方案。

### 下步
确认后直接实现。需要我同时补上其他组件的 hover 效果吗？
```

## 六、与现有系统的关系

```
Custom Instructions  → 日常对话的默认行为（自动生效，轻量）
deep-think 技能      → 复杂问题的深度分析（按需调用，结构化）
CLAUDE.md            → 项目上下文（代码场景自动读取）
```

判断标准：如果你觉得自己需要"好好想想这个问题"，就调 deep-think。如果只是随口问一句，Custom Instructions 的默认行为就够了。

---

## 七、限制

- 只做分析和推荐，不做决策。最终决定权在用户
- 推演完成后用户可以选择让 AI 直接执行，也可以另开任务
- 非代码分析基于训练知识，重要决定请自行验证
- 代码场景依赖 CLAUDE.md 的质量。没有 CLAUDE.md 时用通用模式，推演深度会降低
