# 项目完整生命周期指南（通用模板）

> 本文档是 AI 协作开发的"思考地图"。适用于任何软件项目。
> 每当开始一个新功能、新模块、或重构时，先通读对应章节，确认没有遗漏再动手。
>
> **核心原则：**
> 1. 不是所有功能都是"实体管理"，不同类型有不同的完整思考链
> 2. 功能很少是孤立的，它必然与实体/其他功能交叉，必须查交叉点
> 3. 项目需要的不仅是"把要求的功能做完"，还有"发现还没想到的功能"
> 4. 需求发现不能只靠"自己想"，必须主动搜索和借鉴真实项目

---

## 一、功能类型识别：四类功能，四种思考框架

| 功能类型 | 核心特征 | 典型示例 | 用哪个框架 |
|---------|---------|---------|-----------|
| **实体管理型** | 管理一类数据对象的完整生命周期 | 照片、预设、订单、用户、商品 | 第二章 |
| **工具/操作型** | 对数据执行某种变换/操作，本身不是实体 | 搜索、筛选、排序、导出、批处理 | 第三章 |
| **系统/流程型** | 跨实体、跨模块的机制/流程 | 撤销/重做、同步、导入导出流程、通知、权限 | 第四章 |
| **展示/视图型** | 以不同方式呈现已有数据 | 对比视图、统计仪表盘、时间线、地图视图 | 第五章 |

**关键：** 一个功能常常同时属于多类。比如"导入照片"既是系统流程，又是实体 Create 入口，还触发去重（工具型）。必须识别所有所属类型，每类都套对应框架。

---

## 二、实体管理型功能（CRUD 生命周期）

> 适用于：照片、预设、相册、订单、商品、用户、文档、标签……
> 核心问题：这个"东西"从出生到死亡经历什么？

### 2.1 实体识别

启动项目时，第一步是列出所有数据实体：

- 这个系统管什么"东西"？
- 这些"东西"之间是什么关系？
- 每个"东西"从出生到死亡经历哪些状态？

### 2.2 实体完整生命周期 Checklist

| 操作 | 自问 |
|------|------|
| **Create** | 用户怎么创建？表单？向导？从模板创建？ |
| **Import** | 从哪导入？文件？URL？粘贴？其他系统？ |
| **Read/View** | 列表视图 + 详情视图？还是只有一种？ |
| **Update/Edit** | 哪些字段可编辑？内联编辑？弹窗编辑？ |
| **Delete** | 单个+批量？确认弹窗？软删除还是硬删除？级联影响？ |
| **Export** | 导出什么格式？单导+批量？ |
| **Duplicate/去重** | 什么时候检测？导入时？手动？自动提示还是静默去重？ |
| **Batch** | 批量应用/删除/移动/导出？多选后在哪触发？ |

### 2.3 实体关系地图

| 关系类型 | 自问 |
|---------|------|
| 一对多 | 父包含子列表，可添加/移除子项？ |
| 多对多 | 需要关联表/中间实体，可双向查询？ |
| 依赖 | 删除父实体时子实体怎么办？级联删除/孤立/阻止删除？ |
| 引用 | 原文件被删/移动后引用怎么处理？ |

### 2.4 实体状态流转

```
草稿 → 发布 → 归档
待处理 → 处理中 → 已完成
正常 → 已修改 → 已应用
```

自问：每个状态之间怎么转换？自动还是手动？什么条件？哪些操作只在特定状态允许？

## 三、工具/操作型功能（使用场景链）

> 适用于：搜索、筛选、排序、导出、格式转换、批处理、分析……
> 核心问题：用户在什么场景下使用这个工具？使用前中后各需要什么？

### 3.1 工具型功能的完整思考链

```
触发入口 → 配置选项 → 执行过程 → 结果展示 → 结果后续操作
```

#### ① 触发入口

- 用户从哪里触发？（菜单/按钮/快捷键/自动触发）
- 什么条件下可以触发？（有选中项？有数据？有权限？）
- 什么条件下入口应该隐藏/禁用？

#### ② 配置/输入

- 需要用户提供什么参数？（关键词/范围/格式/阈值）
- 有合理的默认值吗？（用户可以不配直接用）
- 配置界面够简洁吗？（常用一键，高级折叠）
- 上次的配置要记住吗？（持久化偏好）

#### ③ 执行过程

- 即时完成还是需要等待？
- 如果等待：进度怎么展示？可以取消吗？
- 失败了怎么办？部分成功怎么办？

#### ④ 结果展示

- 结果在哪里展示？（替换当前/侧边面板/新视图/弹窗）
- 结果需要高亮/分组/排序吗？
- 无结果时怎么提示？

#### ⑤ 结果上的操作

- 结果上能做什么？（打开/选中/再次筛选/导出结果集）
- 操作后状态保持还是清除？
- 结果可以保存吗？（保存条件/保存结果）

### 3.2 搜索功能 Checklist（示例展开）

| 环节 | Checklist |
|------|-----------|
| **触发** | 搜索框常驻？快捷键唤出？（⌘F）？位置？ |
| **输入** | 即时搜索还是按回车？模糊/精确/正则？支持哪些字段？ |
| **范围** | 当前视图内？全部数据？可切换范围？ |
| **过程** | 即时，大数据量需防抖 |
| **结果** | 原列表过滤/高亮？新视图？结果数量提示？ |
| **无结果** | "没有匹配，尝试其他关键词" + 建议 |
| **自动补全** | 输入时建议？历史搜索？热门？ |
| **历史** | 保留搜索历史？保存常用搜索条件？ |
| **结果操作** | 全选结果？批量操作？导出结果集？ |
| **清除** | 一键清除？Esc 退出？ |

### 3.3 筛选功能 Checklist

| 环节 | Checklist |
|------|-----------|
| **条件** | 单条件？多条件组合？AND/OR？ |
| **预设** | 常用筛选组合可保存？一键切换？ |
| **反馈** | 结果数量即时显示？各筛选项匹配数？ |
| **清除** | 一键清除所有？单个条件独立移除？ |
| **持久化** | 跨视图保持？跨会话记忆？ |

### 3.4 导出功能 Checklist

| 环节 | Checklist |
|------|-----------|
| **对象** | 选中项？筛选结果？全部？ |
| **配置** | 格式/质量/字段选择？ |
| **命名** | 自动命名规则？用户自定义？ |
| **位置** | 默认位置？保存对话框？记住上次？ |
| **过程** | 进度条？预估时间？可取消？ |
| **完成** | Toast？自动打开文件夹？ |
| **失败** | 磁盘不足/权限不足/部分失败？ |

## 四、系统/流程型功能（机制链条）

> 适用于：撤销/重做、同步、导入流程、通知系统、权限系统……
> 核心问题：这个机制穿过哪些模块？每个模块需要做什么配合？

### 4.1 系统型功能的完整思考链

```
触发条件 → 机制范围 → 数据结构 → 各模块配合点 → 异常恢复 → 用户感知
```

#### ① 触发条件

- 什么时候触发？（用户操作/定时/事件驱动）
- 有开关可以关闭吗？

#### ② 机制范围

- 影响哪些实体/操作？
- 哪些操作在范围内？哪些不在？不在的要明确标注

#### ③ 数据结构

- 需要什么额外数据支撑？（撤销栈、同步日志、通知队列）
- 数据量限制？何时清理？

#### ④ 各模块配合点

- 每个相关模块在什么时机做什么动作？
- 模块间如何通信？新增功能如何自动接入此机制？

#### ⑤ 异常恢复

- 机制本身失败怎么办？（同步冲突？撤销失败？）
- 能降级运行吗？

#### ⑥ 用户感知

- 用户需要知道机制在运行吗？
- 什么时候让用户感知？什么时候静默？

### 4.2 撤销/重做 Checklist（示例展开）

| 环节 | Checklist |
|------|-----------|
| **范围** | 哪些操作可撤销？不可撤销的需标注 |
| **栈深度** | 最多多少步？内存限制？ |
| **数据结构** | Command 模式？Snapshot 模式？ |
| **触发** | ⌘Z / ⌘⇧Z / 菜单 |
| **UI 反馈** | 撤销按钮灰显/可用？Toast？ |
| **跨视图** | 在A视图中撤销B视图做的操作？状态同步？ |
| **持久化** | 关闭应用后保留？清空？ |
| **不可撤销** | 物理删除等操作需特殊确认 |

### 4.3 同步功能 Checklist

| 环节 | Checklist |
|------|-----------|
| **范围/方向** | 同步哪些数据？单向上传/双向？ |
| **冲突** | 两端同时修改如何解决？ |
| **断点续传** | 中断后能继续？增量/全量？ |
| **离线** | 离线积压？恢复后自动同步？ |
| **状态** | 同步中/已同步/冲突/失败 用户可见？ |

### 4.4 通知系统 Checklist

| 环节 | Checklist |
|------|-----------|
| **触发** | 什么事件产生通知？ |
| **展示** | 弹窗/状态栏/角标/Toast？ |
| **分组** | 同类合并？最多几条？ |
| **交互** | 点击跳转？可操作通知？ |
| **偏好** | 可关闭某类通知？免打扰？ |

## 五、展示/视图型功能（场景 → 呈现 → 交互）

> 适用于：对比视图、统计仪表盘、时间线、地图视图、看板、日历视图……
> 核心问题：用户在这个视图中要完成什么任务？

### 5.1 展示型功能的完整思考链

```
使用场景 → 数据来源 → 布局结构 → 交互方式 → 与其他视图的切换
```

#### ① 使用场景——用户为什么来这里？

#### ② 数据来源——复用已有数据还是需要计算/聚合？

#### ③ 布局结构——屏幕空间怎么分配？主次关系？

#### ④ 交互方式——只看不动？还是可以直接操作？

#### ⑤ 视图切换——从哪进？怎么出？返回后状态保持？

### 5.2 对比视图 Checklist（示例展开）

| 环节 | Checklist |
|------|-----------|
| **场景** | 找差异？选最佳？对比效果？ |
| **数量** | 2张？最多几张？ |
| **布局** | 并排/叠加/像素级？可切换？ |
| **同步** | 缩放同步？滚动同步？ |
| **操作** | 直接应用预设？删除较差的？ |
| **决策** | 对比完的决策如何体现？ |

### 5.3 统计仪表盘 Checklist

| 环节 | Checklist |
|------|-----------|
| **指标** | 总数/趋势/分布/Top N？ |
| **时间** | 时间范围筛选？对比上期？ |
| **下钻** | 点击下钻到明细？ |
| **刷新** | 实时/手动/定时？ |
| **导出** | 截图？数据导出？ |

## 六、功能生态交叉检查（核心章节）

> 这是解决"跨类型联想"的关键。
> 一个功能从来不是孤岛——它必然与实体、其他功能、用户场景交叉。
> 实现任何功能前，必须做三轮交叉检查。

### 6.1 第一轮：实体穿透检查

**原则：** 每个功能都必然触碰某些实体。检查功能对每个触碰实体的所有影响。

**方法：** 列出功能涉及的所有实体，对每个实体问三个问题：

| 问题 | 说明 |
|------|------|
| 这个功能**读取**了什么实体？ | 读取后要展示吗？展示格式？ |
| 这个功能**修改**了什么实体？ | 修改哪些字段？修改可撤销吗？ |
| 这个功能**创建/删除**了什么实体？ | 级联影响？确认弹窗？ |

### 6.2 第二轮：功能关联检查

**四类功能关联模式：**

| 关联模式 | 含义 | 检查 |
|---------|------|------|
| **前置依赖** | 用A必须先有B | A的入口是否需要B先完成？ |
| **后置触发** | 用完A会触发B | A完成后自动触发什么？ |
| **互斥关系** | A和B不能同时用 | 同时存在时谁优先？ |
| **增强关系** | A+B > A或B | A和B组合能产生更强体验吗？ |

**检查方法：** 新功能 A 实现后，对已有功能每个 B 问上述四个问题。

### 6.3 第三轮：场景回溯检查

**方法：** 为新功能填写以下固定检查表：

```
场景回溯检查表（每个新功能必填）

功能名：________________

1. 用户在什么情境下需要这个功能？
2. 使用这个功能的前一步是什么？（用户从哪来？）
3. 使用这个功能的每一步操作是什么？
   Step1: ___  Step2: ___  Step3: ___ ...
4. 每一步完成后的反馈/下一步引导是什么？
5. 使用完这个功能的后一步是什么？（用户去哪？）
6. 链路断裂点：有没有哪一步用户不知道接下来怎么做？
7. 反馈缺失点：有没有哪一步操作后没有明确反馈？
8. 衔接缺失点：这个功能和前后功能之间的跳转/状态传递有断吗？
```

## 七、需求发现：从自己想 → 搜索真实项目 → 借鉴学习

> **核心转变：** 旧方法是"脑力扫描"——靠自己的经验和想象列需求，永远不够全面。
> 新方法是"AI 主动调研"——搜索真实项目、读取竞品源码和文档、从真实世界中提取需求。
>
> 需求发现分五层，前三层靠推理（快但可能有盲区），后两层靠调研（慢但全面权威）。

### 7.1 第一层：实体驱动发现（推理，快）

**方法：** 拿出实体清单，对每个实体跑一遍"必然需要"列表：

| 必然需要 | 为什么 |
|---------|--------|
| 搜索此实体 | 数据多了就必须能找到 |
| 筛选此实体 | 按属性缩小范围是基本需求 |
| 排序此实体 | 不同任务需要不同顺序 |
| 批量操作此实体 | 逐个操作效率不可接受 |
| 统计此实体 | 用户想知道"我有多少""分布如何" |
| 最近操作此实体 | 用户想快速回到上次操作的对象 |
| 收藏/标记此实体 | 用户需要区分重要/不重要 |
| 对比同类实体 | 用户需要决策"选哪个" |

**AI 行为要求：** 每次新增实体时，必须跑一次这个扫描，结果填入 AGENTS.md。

**局限：** 只能发现"这类实体必然需要什么"，无法发现"这类项目还有你没想过的功能类型"。

### 7.2 第二层：场景驱动发现（推理，中）

**方法：** 写 N 条典型用户旅程，标注每一步的完成状态：

```
旅程名称：________________
用户角色：________________

用户旅程：
Step 1: ___  [✅/⚠️/❌]
Step 2: ___  [✅/⚠️/❌]
...

缺失环节分析：
| 缺失环节 | 说明 | 优先级 |
```

**局限：** 只能发现在你想象到的场景中的缺失，无法发现你根本没想到的场景。

### 7.3 第三层：功能域穷举（推理 + 搜索，关键提升）

> 这一层解决"我不知道我不知道"的问题。
> 方法是：定义项目的功能域，对每个域穷举常见功能。

**方法：**

1. **定义项目的功能域**——这个项目属于什么领域？涉及哪些功能域？

   例：照片管理应用的域 = 文件管理 + 图像处理 + 编辑/预设 + 组织/分类 + 导入导出 + 元数据 + 性能/缓存

2. **对每个功能域，搜索该域的常见功能清单**

   搜索 prompt：
   ```
   搜索并列出 [功能域] 类软件通常具备的全部功能清单，
   按核心/常用/进阶/专家分级。
   参考 [具体竞品名称] 的功能列表。
   ```

   例如对"图像处理"域搜索，可能会发现你从未想到的：
   - 直方图显示
   - 色彩空间转换
   - 批量水印
   - 接触片/联系表（Contact Sheet）
   - HDR 合并

3. **把搜索到的功能清单与已有功能做差集**，得到"域内有但项目没有"的功能

**AI 行为要求：** 项目启动时，必须定义功能域并对每个域执行一次穷举搜索。

**关键：** 7.1 和 7.2 是"从已有实体/场景出发找缺口"，7.3 是"从功能域出发找盲区"——方向相反，互补。

### 7.4 第四层：竞品深度调研（搜索 + 源码分析，全面）

> 这一层解决"别人做了什么而我完全不知道"的问题。
> 不是简单列个功能清单，而是真正去研究竞品怎么做的。

**方法：**

#### 步骤 1：确定调研目标

```
竞品调研目标表

| 竞品名称 | 类型 | 调研原因 | 优先级 |
|---------|------|---------|--------|
|         | 商业软件 |  | 高/中/低 |
|         | 开源项目 |  | 高/中/低 |
|         | 开源项目 |  | 高/中/低 |
```

**选择原则：** 
- 至少 1 个商业软件（代表行业标杆的完整功能集）
- 至少 1 个同类开源项目（可以读源码学习实现方案）
- 至少 1 个跨界参考（非同类但解决了相似问题的项目）

#### 步骤 2：功能清单对比

搜索竞品的功能文档/产品页面，提取完整功能列表：

```
搜索 prompt：
列出 [竞品名称] 的完整功能清单，包括核心功能和边缘功能，
按菜单/模块分组。特别关注 [我们的项目名] 没有的功能。
```

#### 步骤 3：开源项目源码学习

如果有同类开源项目，直接读取其源码结构：

```
调研 prompt：
搜索 GitHub 上与 [项目领域] 相关的星标最高的开源项目，
读取其目录结构和 README，列出它的功能模块划分和核心特性。
重点关注：它的实体有哪些？每个实体有哪些操作？有什么项目没考虑到的功能？
```

**具体操作：**

## 八、交叉关注点（所有功能都适用）

### 8.1 入口三角

每个操作必须至少出现在以下三处之一：

| 入口 | 适合 | 特点 |
|------|------|------|
| 菜单栏 | 全局操作、低频但重要 | 可浏览发现 |
| 工具栏按钮 | 高频、与当前视图相关 | 直观可见 |
| 右键菜单 | 上下文相关操作 | 紧密关联、不打断流程 |
| 快捷键 | 专家用户高频操作 | 效率最高、可发现性差 |

**原则：** 有 API 但上面三处都没有入口 = 对用户来说这个功能不存在。

### 8.2 状态三角

每个列表/网格/面板都可能有三种状态：

| 状态 | 需要 |
|------|------|
| 空状态 | 引导文案 + 行动按钮 |
| 加载状态 | 骨架屏/进度条/Spinner |
| 错误状态 | 错误信息 + 重试按钮 |

空状态文案原则：告诉为什么空 + 告诉能做什么 + 提供行动按钮。

### 8.3 破坏性操作保护

- 弹确认对话框
- 对话框说明具体后果，不只是"确定吗？"
- 默认按钮是"取消"
- 条件允许时提供撤销(Undo)

### 8.4 操作反馈

| 操作耗时 | 反馈方式 |
|---------|---------|
| <1s 即时 | 元素状态变化 |
| 1~5s 短时 | Toast "已完成"/"失败：原因" |
| >5s 长时 | 进度条+百分比+预估时间 |
| 后台 | 状态栏/角标提示进行中 |

### 8.5 快捷键设计原则

| 层级 | 规则 | 示例 |
|------|------|------|
| ⌘/Ctrl + 字母 | 视图级高频操作 | ⌘S 保存，⌘F 搜索 |
| ⌘/Ctrl + 数字 | 切换主视图 | ⌘1 首页，⌘2 详情 |
| ⌘/Ctrl + Shift + 字母 | 同类次级操作 | ⌘⇧F 高级搜索 |
| Delete / Backspace | 删除选中项 | |
| 方向键 | 列表/详情中导航 | |

### 8.6 八类边界情况

- [ ] 空数据（空列表、首次使用）
- [ ] 大数据（列表超长、文件超大）
- [ ] 依赖缺失（引用的外部资源被删/移动）
- [ ] 资源不足（磁盘满、内存不足）
- [ ] 权限不足（只读、未授权）
- [ ] 数据异常（损坏文件、格式错误、0字节）
- [ ] 并发冲突（同时操作同一数据）
- [ ] 网络问题（离线、超时）

## 十一、UI 质量框架（解决"生成的UI很简陋"）

> **核心转变：** UI 简陋不是因为 AI 不会写 CSS，而是因为三个深层问题：
> 1. 没有设计系统——每次凭感觉写，没有统一的设计令牌
> 2. 没有视觉参考——凭记忆想象，不是看着真实设计模仿
> 3. 没有交互细节 Checklist——hover/过渡/焦点/微交互总是漏

### 11.1 设计系统：不让 AI 每次凭感觉

**原则：** 在写任何 UI 之前，必须先定义设计令牌（Design Tokens），所有组件必须使用令牌，不允许硬编码。

#### 必须定义的设计令牌

| 令牌类别 | 需要定义什么 | 示例 |
|---------|-------------|------|
| **间距阶梯** | 4px 为基础倍数，5~6级 | 4/8/12/16/24/32 |
| **圆角阶梯** | 3~4级 | 4px 小按钮 / 8px 卡片 / 12px 面板 / 999px 胶囊 |
| **阴影层级** | 3~4级 | sm 卡片 / md 弹窗 / lg 模态框 / xl 聚焦 |
| **字体层级** | 4~6级 | 11 辅助 / 13 正文 / 15 小标题 / 18 标题 / 24 大标题 |
| **字号权重** | 字号对应什么粗细 | 13 正文=400 / 13 按钮=500 / 18 标题=600 / 24 大标题=700 |
| **动画时长** | 3级 | 150ms 微交互 / 300ms 过渡 / 500ms 弹出 |
| **动画曲线** | 2~3条 | ease-out 出 / ease-in-out 过渡 / spring 弹跳 |
| **组件高度** | 统一标准高度 | 按钮行高 32/36/40 / 输入框 32 / 工具栏 48 / 侧边栏 220 |

**AI 行为要求：** 
- 项目启动时，在主题/样式文件中定义完整的设计令牌
- 所有组件样式必须引用令牌，不允许 `padding: 14px` 这种非阶梯值
- AGENTS.md 中记录设计令牌的数值

#### 组件样式模板

每个组件的样式对象必须包含以下标准字段（缺失则视为不合格）：

```
每个 UI 组件样式必须检查：

基础样式：
- [ ] 背景色（用主题变量）
- [ ] 文字色（用主题变量）
- [ ] 边框（用 borderLight + 阶梯圆角）
- [ ] 内边距（用间距阶梯值）

交互状态：
- [ ] hover 效果（背景色变化 / 阴影提升 / 边框高亮）
- [ ] active/pressed 效果（缩放0.98 / 背景加深）
- [ ] focus 效果（outline 或 borderFocus + 轮廓偏移）
- [ ] selected 效果（accent 色标记）
- [ ] disabled 效果（opacity 0.5 + cursor not-allowed）

过渡动画：
- [ ] 过渡属性（至少 background, box-shadow, transform, border-color）
- [ ] 过渡时长（用令牌值）
- [ ] 过渡曲线（用令牌值）

排版：
- [ ] 字号（用字体层级）
- [ ] 字重（用字号权重对应表）
- [ ] 行高（1.2 标题 / 1.5 正文）
- [ ] 溢出处理（ellipsis / line-clamp）
```

### 11.3 交互细节 Checklist：不让 UI 停在"能用"

**原则：** "能用"和"好用"之间的差距，全在交互细节。以下 Checklist 必须逐项检查。

#### 11.3.1 静态视觉质量

- [ ] **留白充足** — 组件之间有足够间距（至少 8px），不挤
- [ ] **对齐一致** — 同一行元素垂直居中，同一列元素左对齐
- [ ] **视觉层级清晰** — 标题>正文>辅助信息，字号+字重+颜色三层区分
- [ ] **颜色用法一致** — accent 色只用于主操作/选中态，danger 只用于删除/警告
- [ ] **图标统一** — 同一风格（线性/填充），同一大小，不要混用 emoji 和 SVG
- [ ] **圆角一致** — 同一级别组件用同一个圆角值（不要一个 6 一个 8 一个 10）
- [ ] **阴影有层次** — 层级越高阴影越深（卡片 sm / 弹窗 md / 模态 lg）
- [ ] **文字可读** — 正文至少 13px，辅助信息至少 11px，对比度 ≥ 4.5:1

#### 11.3.2 交互状态完整性

**每个可点击元素必须有以下状态：**

| 状态 | 表现 | 缺失会怎样 |
|------|------|-----------|
| **默认** | 基础样式 | — |
| **hover** | 背景变深 / 阴影提升 / 边框高亮 | 用户不知道能点 |
| **active** | 缩放0.98 / 背景再深一档 | 用户不知道点到了 |
| **focus** | accent 色轮廓 / box-shadow 发光 | 键盘用户找不到焦点 |
| **disabled** | opacity 0.5 + cursor not-allowed | 用户不知道为什么不能点 |
| **selected** | accent 背景或边框 | 用户不知道哪个被选中了 |
| **loading** | spinner / 脉冲动画 / 置灰 | 用户不知道是否在处理 |

#### 11.3.3 过渡动画

**必须加动画的交互：**

| 交互 | 动画 | 时长 |
|------|------|------|
| 按钮悬停 | 背景色渐变 | 150ms |
| 面板展开/收起 | 滑动+fade | 300ms |
| 模态框弹出 | scale(0.95→1) + fade | 300ms ease-out |
| 卡片选中 | 边框色渐变 | 200ms |
| 内容切换 | fade in/out | 200ms |
| 列表项进入 | stagger fade + translateY(8→0) | 每项50ms间隔 |
| Toast 出现 | translateY(-20→0) + fade | 300ms |
| Toast 消失 | fade out | 200ms |

**禁止：** 瞬间切换无过渡——这是"简陋感"的最大来源。

#### 11.3.4 微交互（从能用变好用的关键）

| 场景 | 微交互 |
|------|--------|
| 收藏按钮点击 | 星星放大→弹回 + 闪烁一次 |
| 评分切换 | 星星依次点亮动画 |
| 预设应用 | 照片做一次短暂的亮度脉冲（flash）表示"已应用" |
| 删除确认 | 被删项淡出+缩小 → 列表平滑合拢 |
| 拖拽排序 | 拖起项轻微放大+阴影加深 → 放入位置留出空位 |
| 进度完成 | 进度条到100%后短暂变绿+消失 |
| 空状态 | 插图 slowly floating (translateY 上下微动) |

#### 11.3.5 响应式与可访问性

- [ ] 窗口缩小到 800px 时，侧边栏自动折叠
- [ ] 窗口缩小到 600px 时，工具栏文字按钮变图标按钮
- [ ] Tab 键可以在所有可交互元素间聚焦
- [ ] 聚焦元素有明显的 focus ring（不只是浏览器默认）
- [ ] aria-label 覆盖所有图标按钮
