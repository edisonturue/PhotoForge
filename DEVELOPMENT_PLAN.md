# AI 项目构建指导 — 通用开发流程规范

> **本文档适用于任何软件项目。** 它不是某个项目的开发计划，而是 AI 在接到开发任务时必须遵循的思考框架和执行流程。
>
> **核心问题：** 用户描述的需求几乎永远是不完整的。一个"做一个按钮"的需求背后，藏着状态管理、错误处理、边界情况、后续维护等一整条链路。AI 的职责不是只完成字面上的请求，而是**补全用户没想到的部分**。

---

## 一、接到需求后的第一件事：不是写代码，是想清楚

### 1.1 需求拆解三问

收到任何一个功能需求，先问自己：

```
Q1: 这个功能的完整生命周期是什么？
    — 从触发到结束，中间经历哪些状态？有哪些分支路径？

Q2: 用户没说但必然需要的是什么？
    — 加载态？空态？错误态？权限？性能？数据持久化？

Q3: 这个功能和其他模块怎么交叉？
    — 它会影响哪些已有功能？哪些已有功能需要适配它？
```

### 1.2 功能不是孤岛

任何一个功能都至少属于以下四种类型中的一种或多种：

| 类型 | 核心问题 | 典型遗漏 |
|------|---------|---------|
| **实体管理型** | 这个"东西"从创建到删除经历什么？ | 批量操作、去重、导入导出、级联删除 |
| **工具/操作型** | 用户在什么场景下用它？用之前/之中/之后各需要什么？ | 配置持久化、执行进度、结果后续操作 |
| **系统/流程型** | 跨模块的机制怎么运转？ | 撤销/重做、错误恢复、并发处理、状态同步 |
| **展示/视图型** | 数据以什么方式呈现？不同数据量下表现如何？ | 空态、加载态、大量数据时的性能、响应式 |

**关键规则：** 你不能只做"默认态"。每个 UI 组件必须覆盖至少这 7 种交互状态：

```
默认态 → hover → active → focus → disabled → selected → loading
```

如果你只做了默认态就交付，等于只完成了 1/7。

## 二、先搜后造：每个功能都要走调研流程

### 2.1 为什么要搜

```
你以为是新问题 → 其实世界上已经有 100 个人解决过了
你以为要从零写 → 其实 npm 上有现成的库
你以为要设计 UI → 其实竞品已经验证过最佳实践
```

**不搜就写 = 重复造轮子 + 大概率造得更差。**

### 2.2 调研执行流程

对每一个功能，按以下顺序执行：

```
Step 1: 搜索 npm / PyPI / crates.io 等包管理器
        关键词：功能名 + 框架名（如 "react undo redo" "python image histogram"）
        目的：找到可直接安装使用的库

Step 2: 搜索 GitHub
        关键词：功能名 + 技术栈（如 "electron photo editor" "vue dashboard"）
        按 Star 数排序，找最成熟的项目
        目的：找到可参考架构和实现的开源项目

Step 3: 搜索竞品
        找到同领域的商业产品或知名开源产品
        用 Playwright/浏览器打开它们的页面，截图记录 UI
        目的：获取视觉参考和交互模式

Step 4: 搜索设计规范
        Apple HIG / Material Design / Ant Design 等
        目的：获取间距、动画、交互状态等标准参数
```

### 2.3 调研结果分类

调研完成后，将每个功能归入三类之一：

| 分类 | 判断标准 | 做法 |
|------|---------|------|
| **直接复用** | 有成熟库，API 适配当前技术栈，维护活跃 | 安装库，按文档集成 |
| **参考借鉴** | 有开源项目实现了类似功能，但不能直接用 | 阅读源码，借鉴架构和设计思路，自行实现 |
| **自行开发** | 无现成方案，或业务逻辑高度定制 | 从零开发，但参考竞品的 UX 设计 |

**必须记录调研结果。** 即使决定自行开发，也要写清楚"参考了什么、借鉴了什么、为什么不能直接用"。这决定了代码质量的上限。

## 三、补全用户没想到的：功能完整性 Checklist

### 3.1 每个功能必须回答的 15 个问题

用户只说了"做一个 X"，但你必须想清楚：

**触发层：**
1. 用户从哪里触发这个功能？（按钮/菜单/快捷键/自动触发）
2. 什么条件下可以触发？什么条件下应该禁用？
3. 触发时需要什么前置条件？（选中了数据？有权限？有网络？）

**执行层：**
4. 执行是即时完成还是需要等待？
5. 如果需要等待，进度怎么展示？可以取消吗？
6. 执行过程中出错怎么处理？用户看到什么？

**结果层：**
7. 成功后展示什么？需要确认吗？
8. 结果可以撤销吗？如果可以，撤销的粒度是什么？
9. 结果会影响其他模块吗？需要同步更新吗？

**数据层：**
10. 数据存在哪里？内存/本地文件/远程数据库？
11. 数据格式是什么？需要持久化吗？
12. 数据量大了怎么办？需要分页/虚拟滚动/缓存吗？

**边界层：**
13. 并发操作会冲突吗？（同时编辑同一条数据）
14. 数据被外部修改了怎么办？（文件被其他程序删除）
15. 空数据/异常数据时显示什么？

### 3.2 组件交互状态完整性

对于任何 UI 组件，不能只写默认态。必须覆盖：

```
┌─────────┐    hover    ┌─────────┐   active   ┌─────────┐
│ 默认态   │ ─────────→ │ 悬停态   │ ────────→ │ 按下态   │
└─────────┘            └─────────┘            └─────────┘
                            │
                         focus
                            ↓
                       ┌─────────┐
                       │ 聚焦态   │
                       └─────────┘

特殊状态：
  disabled → 禁用态（灰色，不可交互）
  selected → 选中态（高亮/边框变化）
  loading  → 加载态（骨架屏/旋转图标）
```

**每个状态之间的切换必须有过渡动画。** 瞬间切换是"简陋感"的最大来源。

### 3.3 动画与过渡的基本要求

```
按钮 hover     → 150ms 颜色/阴影过渡
面板展开/收起  → 250ms 高度/透明度过渡
模态框弹出     → 200ms 缩放+透明度（从 0.95→1）
视图切换       → 300ms 淡入淡出
列表项删除     → 200ms 缩小+透明 → 合拢
Toast 出现     → 250ms 从顶部滑入
Toast 消失     → 200ms 向上滑出
```

所有动画使用统一的缓动曲线，不要每个组件自己发明。

## 四、想在用户前面：后续问题的预判

### 4.1 功能做完后的"第 N 天问题"

一个功能上线后，用户迟早会遇到这些问题。你要提前想好：

| 问题类型 | 例子 | 应对 |
|---------|------|------|
| **数据膨胀** | "照片越来越多，列表卡了" | 虚拟滚动、分页、懒加载 |
| **误操作** | "我不小心删了/改了" | 撤销/重做、回收站、确认弹窗 |
| **并发冲突** | "两个人同时编辑" | 乐观锁、最后写入胜出提示 |
| **数据迁移** | "我升级了版本，旧数据格式不对" | 版本号 + 迁移脚本 |
| **性能退化** | "数据多了以后导出很慢" | 后台任务 + 进度条 + 可取消 |
| **外部变化** | "文件被其他程序删了" | 缺失文件检测 + 修复建议 |
| **配置丢失** | "重装后设置全没了" | 配置导出/导入 + 云同步 |
| **权限变化** | "磁盘空间满了" | 写入前检查 + 友好提示 |

### 4.2 每个实体的完整 CRUD 不只是四个按钮

创建/读取/更新/删除只是起点。完整的实体生命周期：

```
Create → Read → Update → Delete
  │       │       │        │
  ├ Import│  Filter│  Batch  ├ Soft Delete
  ├ From   │  Sort  │  Export ├ Hard Delete
  │ Template│ Search│  Copy   ├ Cascade Delete
  └ Clone  │  Detail│  Move   └ Archive
            │  Compare│
            │  Group  │
            └ Stats   │
```

当你实现了一个实体的 Create，必须检查：它的 Read 是否支持列表+详情？Update 是否支持内联+弹窗？Delete 是否支持单个+批量+级联？有没有导入/导出？有没有搜索/筛选？有没有统计？

### 4.3 功能间的交叉影响矩阵

每新增一个功能，画出它与已有功能的交叉点：

```
新功能 A  ×  已有功能 1 → 是否需要联动？
新功能 A  ×  已有功能 2 → 是否需要联动？
新功能 A  ×  已有功能 3 → 是否需要联动？
...
```

典型交叉场景：
- 新增"标签"实体 → 搜索功能需要支持按标签筛选 → 导出功能需要支持按标签批量导出 → 统计功能需要显示标签分布
- 新增"撤销"功能 → 所有修改数据的操作都需要纳入历史栈 → 快捷键需要绑定 Cmd+Z → 移动端需要适配手势

## 五、执行纪律：开发阶段的强制检查点

### 5.1 开发前（Phase 0：调研）

```
⛔ 拦截条件：未完成调研不允许写任何代码。

必须完成：
□ 搜索了 npm/PyPI 上是否有现成库
□ 搜索了 GitHub 上的开源参考项目
□ 截图了竞品的 UI 作为视觉参考
□ 查阅了相关设计规范（HIG/Material/Ant）
□ 将调研结果整理为决策记录
□ 确认了"复用/借鉴/自研"的分类
```

### 5.2 开发中（Phase 1-3：设计→实现→集成）

```
每个 Task 完成后必须检查：
□ 代码编译通过
□ 单元测试通过
□ 默认态 + hover + active + focus 态都已实现
□ 空态 / 加载态 / 错误态已处理
□ 过渡动画已添加
□ 快捷键已绑定（如果适用）
□ i18n 文案已添加（如果项目有多语言）
□ 与已有功能的交叉点已处理
```

### 5.3 开发后（Phase 4：验证）

```
⛔ 拦截条件：未完成验证不允许标记完成。

必须完成：
□ 截图验证 UI 效果
□ 对照交互状态 Checklist 逐项确认
□ 测试边界情况（空数据/大数据/异常数据）
□ 测试错误恢复（断网/文件缺失/权限不足）
□ 确认没有引入新的技术债
```

## 六、需求描述不全时的处理策略

### 6.1 用户说"做一个 X"，但 X 远比听起来复杂

```
用户："做一个照片管理功能"
你想：
  - 照片怎么导入？单张/批量/文件夹/拖拽？
  - 导入后存哪里？本地/远程？
  - 支持什么格式？RAW？HEIC？
  - 怎么浏览？网格/列表/时间线？
  - 怎么组织？文件夹/相册/标签/收藏？
  - 怎么搜索？文件名/元数据/内容？
  - 怎么编辑？裁剪/滤镜/参数调整？
  - 怎么导出？格式/质量/尺寸/批量？
  - 怎么删除？单个/批量/回收站？
  - 大量照片时性能怎么办？
  - 数据备份/恢复怎么做？
```

**不要等用户说全了再做。** 你要根据领域知识和竞品调研，主动补全缺失的部分。用户没说不代表不需要。

### 6.2 用户描述了一个功能，但没说边界情况

```
用户："点击按钮导出照片"
你要主动补全：
  - 正在导出时再点怎么办？（禁止重复触发 / 队列化）
  - 导出到没有写入权限的目录怎么办？
  - 磁盘空间不足怎么办？
  - 导出过程中应用崩溃了怎么办？
  - 导出 1000 张照片需要进度条吗？
  - 导出完成后要通知用户吗？自动打开文件夹？
  - 导出的文件命名规则是什么？
```

### 6.3 用户描述了一个 UI，但没说交互细节

```
用户："做一个搜索框"
你要主动补全：
  - 输入时实时搜索还是按回车搜索？
  - 搜索结果怎么展示？下拉列表还是筛选主视图？
  - 有没有搜索历史？
  - 有没有自动补全/建议？
  - 搜索无结果时显示什么？
  - 搜索框在不同屏幕尺寸下怎么响应？
  - 有没有快捷键聚焦搜索框？（如 Cmd+K / Ctrl+K）
```

## 七、技术债的预防与管理

### 7.1 什么会变成技术债

```
"先用 useState 顶一下"          → 状态分散，后面改不动
"这个类型先用 any"              → 类型系统形同虚设
"硬编码的值后面再提取"           → 散落在 50 个文件里
"这个组件先写大一点"             → 2000 行的 God Component
"先不写测试，后面补"             → 后面永远不会补
"先不管性能，能跑就行"           → 用户量上来后崩溃
```

### 7.2 每次写代码时的自检

```
□ 这个常量应该提取到统一配置吗？
□ 这个组件是不是职责太多？需要拆分吗？
□ 这个状态应该提升到全局 store 吗？
□ 这个类型定义够精确吗？有没有用 any？
□ 这个错误处理够完善吗？用户能看到友好提示吗？
□ 这个重复代码应该提取为公共函数/组件吗？
```

## 八、验证标准：功能完成的定义

一个功能不是"代码写完了"就算完成。完成的定义是：

### 8.1 功能完整性

```
□ 核心功能可用（Happy Path）
□ 所有交互状态已实现（7 种）
□ 空态/加载态/错误态已处理
□ 过渡动画已添加
□ 快捷键已绑定（如果适用）
□ 与已有功能的交叉点已处理
```

### 8.2 代码质量

```
□ 编译无错误
□ 测试通过
□ 无 any 类型（或有明确注释为什么用 any）
□ 无硬编码的魔法数字/字符串
□ 公共逻辑已提取
□ 新增类型已定义
```

### 8.3 用户体验

```
□ 截图验证了 UI 效果
□ 操作反馈及时（点击→变化 < 100ms）
□ 大数据量下不卡顿
□ 错误信息对用户友好
□ 文案已添加到 i18n（如果项目有多语言）
```

### 8.4 可维护性

```
□ 后续修改这个功能的人能看懂代码
□ 数据格式变化时有迁移路径
□ 功能可以安全地禁用/回滚
□ 没有引入新的技术债
```

## 九、参考资源的使用方法

### 9.1 竞品调研

```
目的：不是抄袭，是理解"这个行业的人怎么解决这个问题"

做法：
1. 找到 2-3 个同领域最成熟的产品
2. 用 Playwright 打开它们的页面
3. 截图记录：布局、交互流程、状态切换、错误处理
4. 分析：哪些设计是行业共识？哪些是差异化？
5. 决定：采用共识设计，差异化部分自行判断
```

### 9.2 开源项目参考

```
目的：不是 fork，是学习"有经验的开发者怎么组织代码"

做法：
1. 找到同技术栈、同领域的高 Star 项目
2. 阅读目录结构 → 理解架构分层
3. 阅读核心模块 → 理解状态管理和数据流
4. 阅读测试 → 理解质量标准
5. 记录可借鉴的模式，在自己的项目中实现
```

### 9.3 设计规范参考

```
目的：不是照搬，是确保"交互行为符合用户预期"

必查项：
- 间距系统（不要每个组件自己定 px 值）
- 圆角系统（统一的圆角阶梯）
- 阴影系统（层级分明的阴影）
- 字体系统（字号 + 字重 + 行高）
- 动画系统（时长 + 缓动曲线）
- 交互状态（hover/active/focus/disabled 的标准样式）
- 组件尺寸（按钮高度、输入框高度、卡片尺寸）
```

## 十、总结：AI 开发者的行为准则

```
1. 收到需求 → 先想完整生命周期，不是先写代码
2. 动手之前 → 先搜 npm/GitHub/竞品，不是从零开始
3. 用户没说 → 主动补全，不是等用户补充
4. 写代码时 → 想后续问题，不是只做当前功能
5. 交互设计 → 覆盖 7 种状态，不是只做默认态
6. 状态切换 → 加过渡动画，不是瞬间切换
7. 功能完成 → 验证完整性，不是代码跑通就行
8. 每次提交 → 自检技术债，不是积累到不可维护
```

**一句话：不要做"写了就跑"的程序员，要做"写了就不用再回来改"的程序员。**
